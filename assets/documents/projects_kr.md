Resume의 bullet을 뒷받침하는 공개용 case study입니다. 내부 테이블명, 상세 시스템명, 법인명, 금액 효과는 제외하고 문제 정의, 데이터 접근, 직접 수행한 역할, 적용 방식 중심으로 정리했습니다.

## Deep Case Studies

### 사용자 단말 센서 품질 로그(DQA) Data Agent / AI-ready Data 구축

스마트폰 품질 로그에는 부품과 센서에서 발생하는 계측 데이터가 대량으로 쌓입니다. 이 데이터는 디바이스 상태를 점검하고, 다음 제품 라인업의 사양을 정하거나 불량 디바이스의 로그를 분석하는 데 쓰입니다. 문제는 raw log가 JSON 문자열 형태로 들어오고, key별 의미와 단위, 정상 범위, 전처리 방식이 모두 다르다는 점이었습니다. 일부 key는 spec 문서에는 있지만 실제로 거의 수집되지 않았고, 반대로 spec에는 없지만 실제 데이터에는 들어오는 경우도 있었습니다. 하드웨어 개발자가 직접 SQL을 작성하기 어렵고, 분석 조직에 요청하더라도 key 해석 노하우가 흩어져 있어 깊은 분석으로 이어지기 어려운 구조였습니다.

제가 맡은 일은 이 데이터를 Agent가 읽고 분석할 수 있는 AI-ready data로 바꾸는 것이었습니다. 먼저 spec 문서와 실제 인입 데이터의 차이를 비교해 어떤 key가 실제로 분석 가능한지 확인했습니다. 이후 key별 의미, 단위, 전처리 규칙, 값의 범위, 분석 시 주의점을 정리했습니다. Raw log를 직접 조회하지 않아도 되도록 mart table을 설계하고, LLM이 table과 column을 오해하지 않도록 description과 domain knowledge를 작성했습니다. 실제 사용자가 물을 만한 질문 유형도 정의하고, 특정 배터리 온도 구간의 발생 비중, 모델별 평균 온도, 특정 error log 발생률 같은 질문에 대한 Question-SQL pair와 검증용 기준 query를 만들었습니다.

역할 경계도 명확했습니다. SQL 생성, 검증, 실행, 해석 파이프라인은 별도 개발 조직에서 담당했고, 저는 Agent가 틀린 key를 선택하거나 단위를 잘못 해석하거나 raw JSON을 비효율적으로 조회하지 않도록 data/knowledge preparation 영역을 맡았습니다. 하드웨어 담당자와 key 의미를 검증하고, 운영 중 나온 문의와 답변 품질 이슈도 확인해 수정했습니다.

성과는 하드웨어 개발자가 SQL과 Python 없이도 품질 로그를 자연어로 질의할 수 있게 된 점입니다. 사용자는 "모델별 특정 error log 발생률", "국가별 온도 구간 비중", "전체 디바이스 대비 특정 조건 발생 비율"처럼 업무 언어로 질문하고, Agent는 SQL 결과와 함께 key 처리 방식, 단위, 결과 해석을 돌려줍니다. 초기 영역에서 성과를 인정받아 30여개 component 대상으로 확대 진행 중이며, 정식 운영 후 사용자 문의에 따른 개선까지 수행했습니다.

### TV 공정 단계 잠재 불량 예측

TV 제품은 공정을 마친 직후에는 정상으로 보이더라도 이후 사용 과정에서 시장불량으로 나타날 수 있습니다. CS 단계에서 불량을 대응하는 것보다 공정 직후 잠재 불량 단계에서 조기에 잡아내는 것이 비용과 브랜드 측면에서 유리합니다. 이 과제는 그런 방향성에 대한 조직적 공감대에서 시작되었습니다. 제가 맡은 일은 공정 단계에서 이미 수집되는 검계측 데이터와 이후 서비스센터에서 발생하는 불량·수리 이력이 연결될 수 있는지 확인하고, 이를 예측 문제로 바꾸는 것이었습니다.

핵심은 데이터 연결이었습니다. 공정 검계측 데이터와 전세계 서비스센터의 CS 불량·수리 이력은 serial number를 기준으로 연결될 수 있었습니다. 이를 통해 공정 완료 시점의 데이터로 이후 불량 발생 가능성을 예측하는 지도학습 문제를 구성했습니다. 데이터 매핑과 전처리, 의미 단위로 학습 데이터를 묶는 구조 설계, 초기 모델링 코드, 초기 평가 metric 코드를 직접 구현했습니다.

평가 방식도 단순 precision만으로는 충분하지 않았습니다. 불량은 희소하고, 제품 출시 이후 시점에 따라 기본 불량률도 달라질 수 있습니다. 같은 precision이라도 어떤 시점의 baseline과 비교하느냐에 따라 의미가 달라질 수 있었습니다. 그래서 랜덤 샘플링 대비 precision이 몇 배 개선되었는지를 보는 lift 지표를 제안했습니다. 시점별 기본 불량률 차이를 어느 정도 normalize해 모델 성능을 해석하기 위한 장치였습니다.

운영 단계에서는 threshold가 가장 중요했습니다. Positive로 예측된 세트는 aging test를 우선 진행해야 했고, false positive가 너무 많으면 라인 병목이 생깁니다. 따라서 라인에서 실제로 처리 가능한 aging test 물량과 탐지 효과 사이의 균형을 현업과 협의해 threshold를 설정했습니다. 또한 현재 불량이 있는 세트를 잡는 것이 아니라 가까운 시일 내 불량 가능성이 높은 세트를 고르는 문제였기 때문에, 주요 feature와 유사 과거 데이터의 경향을 함께 전달하는 방식도 협의했습니다. 적용 기간 중 관련 불량률은 절반 수준으로 낮아졌습니다.

### BLU Simulator 개발 및 검증

TV의 miniLED 백라이트는 화면 뒤쪽에 배치된 광원이 위치별로 밝기를 다르게 내면서 화면 품질에 영향을 줍니다. 광원 사이의 간격, 광원과 패널 사이의 거리, 광원의 구동 조건에 따라 uniformity, 블루밍, 블록단차 같은 화질 문제가 달라집니다. 신규 백라이트 방식을 도입하는 초기에는 경험과 노하우가 부족했고, 모든 조합을 실물 패널로 제작해 육안 평가하는 방식은 시간과 재료비 부담이 컸습니다.

이 문제를 데이터와 시뮬레이션 문제로 바꾸었습니다. 협업 부서가 촬영한 70여 가지 단일 광원 광분포를 받아 parametric curve로 fitting했습니다. 이후 광원과 패널 사이 간격이 변할 때 curve parameter가 어떻게 변하는지 회귀분석으로 확인했고, 제한된 측정값만으로 다양한 간격의 광분포를 생성할 수 있게 했습니다. 또한 두 개 이상의 인접 광원이 있을 때 광분포가 단순 덧셈으로 중첩되는지 검증해, 광원 배치에 따른 전체 패널 백라이트 시뮬레이션 로직을 만들었습니다.

화질 문제를 수치화하는 지표도 직접 설계했습니다. Uniformity는 화면 전체의 균일도를 판단하는 기준으로 정의했고, 블록단차는 이미지 경계 검출에 쓰이는 소벨 필터를 활용해 정량지표로 만들었습니다. 실제 광원을 실장할 때 생기는 오차 때문에 일부 영역의 지표가 나빠지는 문제가 있었고, 이를 단일 평균값으로 덮어버리면 현업이 믿고 쓰기 어려웠습니다. 그래서 화면 전체와 부분 영역을 함께 볼 수 있는 형태로 여러 지표를 마련했습니다.

최종 결과물은 Python으로 구현한 뒤 PyQt와 packaging 도구를 활용해 Windows 프로그램으로 배포했습니다. 사용자는 선행 패널 개발 부서였습니다. 사용자는 광원 분포, 광원과 패널 사이 간격 범위, 최소 간격, 구동 제약조건 등을 입력하고, 출력으로 세로 간격, 가로 간격, 광원-패널 간격을 포함한 설계 옵션 후보를 확인할 수 있었습니다. 실물 패널 촬영 이미지와 시뮬레이션 밝기 map을 normalize해 비교했을 때 평균 90% 이상 유사도를 확보했고, miniLED 개발 프로세스 적용, 개발 기간 절반 수준 단축, 재료비 70% 이상 감축으로 이어졌습니다.

### Galaxy Smart Charging / Principal Data Scientist 인증 과제

Galaxy 스마트폰에는 일반, 고속, 초고속 충전 옵션이 있습니다. 사용자가 한 번 고속 충전을 켜두면 충전기가 지원하는 한 언제든 고속 충전으로 진행됩니다. 하지만 자는 동안처럼 장시간 충전하는 상황에서는 천천히 충전해도 아침까지 완충이 가능하고, 굳이 빠르게 충전해 배터리 수명 부담을 키울 필요가 없습니다. 문제는 이런 상황을 사용자가 매번 직접 판단하고 설정을 바꾸기 어렵다는 점이었습니다.

이 과제는 삼성전자 DX부문 Data Science Level 4 인증 과정의 2023년 1년 과제로 수행했습니다. 제가 정의한 방향은 디바이스별 최근 로그를 이용해 개별 사용자의 수면 시간대를 파악하고, 장시간 충전 구간에서는 배터리 보호를 위해 완속 충전으로 전환하는 것이었습니다. 사용한 데이터는 충전 시작/종료 시점, 충전 시작/종료 배터리 잔량, 앱 사용 로그였습니다. 별도 복잡한 모델을 쓰기보다 앱 사용 시간 기반 heatmap을 보고, 액티브 사용량이 낮아지는 시간대가 반복되는지를 이용해 수면 패턴을 판단하는 로직을 설계했습니다.

제가 직접 수행한 범위는 문제 정의, 데이터 분석, 수면 시간대 파악 로직 개발, 완속 충전 전환 로직 선행 개발, 사용자 안내와 빠른 고속 충전 전환 UX 방향 제안이었습니다. 실제 Android 코드, 설정 앱 UI, 충전 제어 로직 반영, 제품 테스트는 협업 부서가 담당했습니다. 제품 전체 구현을 혼자 수행했다는 뜻이 아니라, 데이터 기반 선행 로직과 사용자 경험 방향을 만들어 제품 옵션 반영으로 이어지게 한 프로젝트입니다.

이 프로젝트는 Principal Data Scientist 인증의 핵심 근거가 되었고, 제품에 연결된 데이터 사이언스 경험으로 남았습니다. 단순 예측 모델이 아니라 사용자의 행동 패턴을 해석하고, 그 결과가 실제 제품 동작과 사용자 안내로 이어지도록 만든 경험입니다.

### Trade-in Fraud Detection

온라인 자사몰에서는 기존 기기를 제출하는 조건으로 신규 기기 구매 시 trade-in incentive를 즉시 할인해주는 구조가 있습니다. 구매 시점에는 아직 기존 기기를 받기 전이지만, 고객 경험과 구매 전환을 위해 먼저 할인을 적용하고 새 제품을 출고합니다. 문제는 일부 주문이 기존 기기를 제출하지 않고, 이후 회수 절차도 회피하면서 손실로 이어진다는 점이었습니다. 기존 기기가 실제로 도착한 뒤 payback을 제공하는 방식은 사업 구조상 현실적이지 않았기 때문에, 구매 시점에 의심 주문을 탐지하고 주문 담당자가 한 번 더 확인할 수 있게 하는 것이 목표였습니다.

데이터는 주문, 고객, 구매 방식, 제품 수령 방식, 적용 할인, 제출 예정 구형 모델, 과거 구매 이력, 회수 절차 로그 등을 활용했습니다. Label은 기존 기기 미제출과 회수 실패 여부를 기준으로 정의했습니다. 제출 예정 구형 모델의 특성, 수령 방식, 할인 조합, 고객의 과거 주문 패턴, 주문 단위의 여러 신호를 결합해 fraud risk를 설명할 수 있는 feature를 발굴했고, XGBoost 기반 분류 모델을 학습했습니다.

운영 설계에서는 모델 점수만큼 threshold가 중요했습니다. 의심 주문을 너무 많이 보내면 주문 검토 업무가 과도해지고, 너무 적게 보내면 손실을 줄이기 어렵습니다. 따라서 시간에 따라 변하는 probability distribution을 기준으로 threshold를 조정해 탐지 물량을 제어했습니다. 모델은 주기적으로 재학습되도록 했고, positive로 예측된 주문은 dashboard로 전달했습니다. Dashboard에는 판단에 영향을 준 주요 feature, 전체 feature 값, 고객의 과거 주문 정보가 포함되었습니다. 최종 취소 여부는 사람이 판단하는 human-in-the-loop 구조였습니다.

검증은 일부 주문에만 모델을 적용하고, 미적용군의 사후 결과를 활용하는 A/B test 유사 방식으로 진행했습니다. 이렇게 해야 모델이 잡은 주문뿐 아니라 흘려보낸 주문에 대해서도 나중에 성능을 확인할 수 있습니다. 운영 기준 precision은 70% 수준으로 확인했습니다. 즉 positive로 예측한 주문 중 상당수가 실제 fraud에 해당한다는 의미입니다. 금액 효과는 내부성이 강해 공개 문서에서는 제외합니다. 제가 직접 수행한 범위는 EDA, feature 발굴, 모델 학습, threshold 설계, dashboard 제공 정보 정의 등 데이터와 모델링 전반이었습니다.

## Additional Case Studies

### Enterprise Knowledge System / Ontology / 지식 그래프 PoC

상용 LLM을 업무에 활용할 수 있게 되면, 가장 먼저 부딪히는 한계는 모델이 사내 지식을 모른다는 점입니다. 공개 웹에 있는 지식은 잘 설명해도 사내 용어, 시스템 사용법, 데이터 table의 의미, 부서별 convention, 암묵지는 기본적으로 알 수 없습니다. 사용자가 매번 긴 설명을 prompt에 넣는 방식으로는 확장성이 없습니다. 따라서 사내 지식을 LLM이 활용할 수 있는 형태로 정리하고, 필요한 지식을 안정적으로 검색·주입·관리할 수 있는 체계가 필요했습니다.

이 과제에서는 Markdown 기반의 LLM Wiki와 공개 지식 포맷 아이디어를 참고해 사내 지식을 구조화하고, 이를 Ontology, RDF, 지식 그래프로 승격하는 PoC를 수행했습니다. 제가 직접 수행한 일은 Ontology 설계 PoC, RDF 변환 규칙 설계, entity/relation 추출 기준 설계, 기본 지식 그래프 구축, 상충·모호·불일치 지식 탐지 PoC 구현이었습니다. 샘플 지식에 대해 Ontology와 지식 그래프를 만들고, 탐지 예시를 구현해 내부 데모와 발표를 진행했습니다. 다만 LLM Wiki schema 자체를 설계한 것은 아니고, 전체 운영 시스템이 이미 전사 적용된 것도 아닙니다.

이 프로젝트는 사내 지식을 AI-ready하게 만드는 과제입니다. 도메인 담당자가 지식을 작성하고 관리하되, Ontology와 relation rule을 쉽게 다룰 수 있도록 guide, Agent, Skill 형태로 보조하는 구조를 지향합니다. 현재는 PoC와 아키텍처 정의 단계입니다.

### 서비스 자재 / TV 패널 수요예측

TV 패널은 모델과 연식에 따라 구조가 달라 호환이 어려운 경우가 많습니다. 제품 단종 시점에는 향후 서비스 교체에 필요한 패널을 미리 확보해야 합니다. 적게 확보하면 신형 제품 교환이나 환불로 이어질 수 있고, 많이 확보하면 과잉 재고와 폐기 비용이 발생합니다. 이 문제는 단순 수요예측이 아니라 과소 예측과 과다 예측이 모두 비용으로 이어지는 장기 자재 의사결정 문제였습니다.

접근의 핵심은 유사자재를 스펙이 아니라 판매·서비스 수요 패턴으로 정의한 점입니다. 꾸준히 팔리는 라인업과 특정 시기에 집중 판매되는 제품은 이후 서비스 수요의 모양도 달랐습니다. 판매 trend와 서비스 trend를 시간축과 수량축에서 조정하면 유사한 형태가 나타나는 경우가 있었고, 이를 활용해 예측대상자재와 유사한 자재를 매핑했습니다. 저는 문제 정의, 데이터 전처리, 유사자재 매핑 로직 구현, 전체 수량과 월별 trend를 분리해 예측 정확도를 판단하는 아이디어 제안에 깊게 관여했습니다.

시스템화 단계에서는 별도 개발자가 구현을 맡았고, 저는 초기 로직과 관리 역할을 중심으로 수행했습니다. 최종적으로 benchmark 오차율 29.6% 대비 19.1% 수준까지 개선되어 목표 오차율을 달성했고, 현업 시스템 적용과 발명 출원으로 이어졌습니다. 금액 효과는 내부성이 강해 공개 문서에서는 제외합니다.

### Procurement Cost Analysis Simulator

전세계 생산법인에서 쓰는 자재는 종류가 많고, 기능적으로 유사한 자재라도 공급사, 배송 방식, 가격, 리드타임, 수급 가능 수량이 달라질 수 있습니다. 구매 관점에서는 유사 자재의 사양과 단가를 비교해 원가 절감 아이디어를 찾고 싶지만, 데이터 확보와 반복 분석 자체가 큰 병목이었습니다. 특히 분석에 필요한 데이터가 정식 interface로 제공되지 않아, 고도화된 분석 로직보다 데이터 확보가 먼저 해결해야 할 문제였습니다.

저는 방향을 바꿔 데이터 확보 자동화와 분석 도구를 별도로 만들었습니다. Python 기반 Excel 자동화 프로그램으로 필요한 데이터를 확보하고, 확보된 데이터를 여러 사용자가 활용할 수 있도록 Python/PyQt 기반 Windows 프로그램을 만들었습니다. 프로그램은 유사자재 사양과 단가 비교, 가격 trend, 물류, BOM 등 관련 정보 확인, 조건 customizing, interactive 시각화와 report download를 지원했습니다.

로컬 프로그램으로 시작했지만 유용성을 인정받아 이후 웹 구현으로 전환되었고, 제 로직과 코드를 웹 개발용으로 이관했습니다. 이 과제는 모델링보다 업무 병목을 정확히 잡는 것이 중요했던 사례입니다. 기존 분석 업무시간은 91% 수준으로 단축되었습니다.

### Data Request Dashboard / Data Mart

사내 여러 조직에서는 필요한 데이터를 직접 추출하지 못해 분석 조직에 데이터 요청을 보냈습니다. 요청을 처리할 분석가는 제한적이고, 요청 중 상당수는 조건과 dimension만 조금씩 다른 반복 작업이었습니다. 기존에는 분석가가 과거 SQL을 복사해 수정하거나 새로 작성하는 방식으로 대응했기 때문에, 시간이 많이 들고 담당자별 SQL 구조와 결과물 품질도 달라질 수 있었습니다.

저는 최근 2년치 데이터 요청 SQL 약 2,000건을 분석해 반복되는 CTE 구조, 공통 전처리, 조건/집계 패턴을 파악했습니다. 이미 SQL을 실행하는 dashboard platform은 있었기 때문에, 새 platform을 만든 것이 아니라 UI에서 입력한 값을 SQL에 동적으로 반영할 수 있는 template 구조를 설계했습니다. BigQuery, dashboard tool, Jinja template을 활용해 정형 요청을 GUI 기반으로 처리할 수 있게 했고, 약 30개 mart table과 여러 chart를 설계했습니다.

추가로 Python query parser와 검색용 web module, Git 기반 SQL review workflow도 만들었습니다. 데이터 요청 한 건이 끝나면 SQL을 repository로 가져와 구조를 parsing하고 diagram을 생성한 뒤, 다른 분석가가 review할 수 있는 흐름을 정립했습니다. 이 과제는 분석가 조직 내부의 생산성과 결과물 일관성을 높인 프로젝트였습니다.
